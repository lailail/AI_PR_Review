# AI PR Review Assistant

AI PR Review Assistant 是一个面向 GitHub Pull Request 的 AI 代码评审辅助工具。用户输入 GitHub PR 链接后，系统会获取 PR 变更内容，并逐步生成 PR 变更总结、风险代码识别和 Review 建议，帮助开发者提升评审效率与质量。

## 当前状态

当前仓库已经具备从 GitHub PR 获取变更、调用 DeepSeek 生成总结/风险/Review 建议、保存本地历史记录，并进行同仓库历史 PR 对比分析的基础能力。

本阶段已完成：

- GitHub PR URL 解析
- 后端 GitHub API 请求封装
- PR 基础信息获取
- changed files 和 patch 获取
- 前端加载状态、错误状态和结果展示
- GitHub Token 环境变量示例
- DeepSeek API 服务端调用
- AI 生成结构化 PR 变更总结
- 规则预筛选高风险文件
- DeepSeek 生成结构化风险识别结果
- DeepSeek 生成结构化 Review 建议
- 低置信度建议过滤
- Markdown Review 结果复制
- 本地 PR 分析历史记录
- 按仓库归类展示历史分析摘要
- 同仓库历史 PR 对比分析

## 面向真实代码评审的需求理解

真实 Pull Request Review 的核心目标不是“让 AI 多说几句”，而是帮助 reviewer 更快判断这次改动是否安全、是否完整、是否值得合并。围绕这个目标，本项目把 AI 辅助能力拆成三个主要环节：

- **PR 变更总结**：先回答“这次 PR 改了什么、影响哪些模块、测试是否覆盖”，降低 reviewer 阅读大段 diff 的启动成本。
- **风险代码识别**：优先找出可能影响线上质量的变更，例如权限、接口入参、数据库读写、异常处理、配置依赖、测试删除等。
- **Review 建议生成**：把风险证据转换成可执行的 Review 反馈，方便开发者直接定位文件、理解原因并修改代码。

在使用体验上，工具不强制一次性生成所有结果，而是把 GitHub 数据获取、变更总结、风险识别、Review 建议和历史对比分开触发。这样 reviewer 可以先快速看结构化摘要，再决定是否继续调用成本更高的 AI 分析。

## 本地运行

安装依赖：

```bash
npm install
```

复制环境变量示例：

```bash
cp .env.local.example .env.local
```

按需配置 GitHub Token 和 DeepSeek API Key：

```text
GITHUB_TOKEN=your_github_token
DEEPSEEK_API_KEY=your_deepseek_api_key
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-flash
DEEPSEEK_CONTEXT_API_KEY=
DEEPSEEK_CONTEXT_MODEL=deepseek-v4-pro
```

启动开发服务：

```bash
npm run dev
```

启动后访问：

```text
http://localhost:3000
```

## GitHub Token 说明

公开仓库的 PR 可以尝试匿名请求，但 GitHub 匿名 API 限流较低。建议配置 `GITHUB_TOKEN`，提升请求稳定性。

Token 只在服务端读取，用于请求 GitHub REST API，不会暴露到浏览器端。

## DeepSeek API Key 说明

PR 变更总结、风险代码识别和 Review 建议由服务端调用 DeepSeek Chat Completions 接口生成。

真实 API Key 必须只写在本地 `.env.local` 文件中：

```text
DEEPSEEK_API_KEY=your_deepseek_api_key
```

`.env.local` 已在 `.gitignore` 中忽略，不能提交到 GitHub。仓库中只提交 `.env.local.example`，用于说明需要哪些环境变量。

如果其他人下载本项目代码，需要自己复制 `.env.local.example` 并创建本地 `.env.local` 文件，然后填写自己的 API Key。真实 Key 不会也不应该出现在 GitHub 仓库中。

默认配置：

```text
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-flash
DEEPSEEK_CONTEXT_API_KEY=
DEEPSEEK_CONTEXT_MODEL=deepseek-v4-pro
```

前端不会读取 DeepSeek API Key，只会调用本项目的 `/api/ai-summary`、`/api/risk-detection` 和 `/api/review-suggestions` 后端接口。

页面右上角提供“模型配置”入口，用于本地演示时切换 DeepSeek API Key 和模型。用户填写 API Key 后可以点击“获取模型”，系统会调用 DeepSeek `/models` 接口获取该 Key 可用的模型列表，再从下拉框中选择模型。该功能会通过后端写入本地 `.env.local`，只适合本地开发环境；如果部署到公网服务器，不应该开放给未授权用户修改。

历史 PR 对比分析优先使用 `DEEPSEEK_CONTEXT_API_KEY` 和 `DEEPSEEK_CONTEXT_MODEL`。如果没有配置大上下文专用 Key，则回退使用普通 `DEEPSEEK_API_KEY` 和 `DEEPSEEK_MODEL`。这样可以把普通 PR 分析和历史上下文分析分开配置，也方便后续接入其他大模型。

## 本地历史记录说明

系统会在用户分析 PR 后，把本次分析摘要保存到当前浏览器的 `localStorage` 中，并使用 `owner/repo` 作为仓库分类 key 展示历史记录。

历史记录用于体现“上下文理解”的本地积累方式：系统不仅能理解当前 PR 的标题、描述、changed files 和 patch，也能知道当前浏览器之前分析过同一仓库的哪些 PR。页面会按仓库归类历史 PR，并允许用户回看某次 PR 的全部变更内容。

为控制体积和降低敏感代码暴露风险，历史记录不会保存完整仓库代码。系统会保存 AI 总结、风险数量、Review 建议数量、供模型历史对比使用的截断 `patchDigest`，以及供页面回看的 `changedFileSnapshots`。其中 `changedFileSnapshots` 会按文件保存本次 PR 的 diff patch，并对单个超大 patch 做长度保护。清空浏览器缓存或点击页面中的“清空历史”后，这些记录会被删除。

历史记录使用固定 key `ai-pr-review-history:v1` 保存。页面读取历史记录时会清洗本地缓存数据，忽略格式损坏或缺少必要字段的记录，避免浏览器缓存异常影响当前 PR 分析流程。同一个仓库的同一个 PR 重新分析时会覆盖旧记录，历史记录默认最多保留 50 条。

该能力只适合本地演示和单人使用；如果后续要支持团队共享历史上下文，应改为后端数据库、登录权限和加密存储方案。

## 历史 PR 对比分析

当当前 PR 所属仓库存在本地历史分析记录时，页面会提示可启用“结合历史 PR 对比分析”。默认模式仍然是“仅分析当前 PR”，系统不会自动把历史记录提交给模型，用户需要手动切换模式并点击“生成历史对比分析”。

历史对比只会选择同仓库历史记录，并排除当前 PR 自己。后端默认最多使用最近 10 条历史记录，且只使用历史总结、风险、Review 建议和截断后的 `patchDigest`，不会发送完整仓库代码或完整 diff。

历史对比结果会展示：

- 当前 PR 与哪些历史 PR 有关系。
- 当前 PR 可能影响哪些历史功能或模块。
- 是否重复出现类似风险。
- reviewer 建议重点回看的历史 PR 或模块。
- 本次历史上下文分析的置信度和限制说明。

## 产品规划

最终工具会围绕 GitHub PR 完成以下流程：

1. 用户输入 GitHub PR URL。
2. 系统解析 owner、repo 和 PR 编号。
3. 系统通过 GitHub API 获取 PR 元数据、变更文件和 diff。
4. 系统筛选关键上下文，优先关注高风险变更。
5. 系统调用 DeepSeek 生成结构化 PR 总结、风险识别结果和 Review 建议。
6. 页面展示 PR 总结、风险点和可复制 Markdown Review 建议。
7. 浏览器本地保存本次分析摘要，并按仓库归类展示历史记录。
8. 用户可手动启用同仓库历史 PR 对比分析，查看当前 PR 与历史 PR 的关系。

## AI 分析设计思路

系统不会简单地把完整 diff 一次性发送给模型，而是先通过 GitHub API 获取 PR 元数据、文件列表和 diff，再结合文件类型、变更规模、风险关键词、历史分析记录等信息进行分层处理。这样可以在保证上下文足够的同时，减少无关内容对模型判断的干扰。

AI 分析分为三类输出：

- **变更总结**：基于 PR 标题、描述、变更规模、文件列表和关键 patch，生成业务变化、技术变化、测试变化和影响范围。
- **风险识别**：基于规则预筛选信号和关键 patch，输出风险等级、风险类别、文件位置、diff 证据、风险原因和置信度。
- **Review 建议**：复用变更总结和风险识别结果，把证据转成更适合提交到 PR 评论区的建议内容。

风险识别会先通过规则预筛选高风险文件，例如鉴权、接口、数据库、配置、依赖和测试文件变化，再优先把这些文件的 patch 放入模型上下文，以提升分析速度和相关性。

文档类文件会单独降噪处理。README、Markdown 文档或 docs 目录中的内容如果提到 API、Token、配置等关键词，只会标记为“文档说明”，不会和代码文件一样参与高风险排序。

对于可以规则化的典型代码模式，系统会尽量用规则稳定识别。例如自定义 `Error` 类中出现 `super(code)` 这类写法时，会标记为“错误处理”信号，避免完全依赖模型随机识别。

Review 建议生成会优先复用 PR 总结和风险识别结果，让模型围绕已有证据输出建议，避免脱离 diff 的泛泛评论。

本项目不是只面向前端项目。当前实现已经把规则入口设计为可扩展的文件类型和风险信号识别，后续可以继续扩展 Java Spring Boot、Node.js 后端、数据库脚本、Redis、MySQL、RocketMQ、配置中心、CI/CD 等不同技术栈的专项规则。

AI 分析会优先关注：

- 权限与鉴权逻辑
- 数据库读写
- API 入参校验
- 异常处理
- 并发与异步逻辑
- 测试删除或覆盖不足
- 依赖与配置变更
- 消息队列生产与消费逻辑
- 缓存一致性与过期策略
- 事务边界和回滚逻辑
- 配置、密钥和环境变量变更

## 误报与漏报控制

当前风险识别与后续 Review 建议会尽量做到：

- 每条建议绑定文件和行号。
- 风险识别结果重点展示 diff 证据和风险原因。
- Review 建议结果提供具体修复方向。
- 每条建议标记严重级别。
- 每条建议提供置信度。
- 低置信度建议标记为“需要人工确认”。
- 风格类建议降低优先级，避免干扰核心评审。
- 规则预筛选只作为信号，不直接当成最终风险结论。
- 文档文件命中关键词时会降级为说明性信号，减少 README 等文件带来的噪声。
- 缺少 diff 证据的风险不会展示。
- 默认隐藏低置信度 Review 建议，用户可手动显示。
- 复制出的 Markdown 会提示低置信度建议需要人工确认。
- 风险识别为空但 Review 建议存在时，页面会说明 Review 建议仍可能来自 diff、总结和中低风险可维护性线索。

误报控制的重点是避免“看到关键词就下结论”。因此规则预筛选只负责提示可能相关的文件，最终风险仍需要模型结合 diff 证据输出原因；文档类文件、低置信度建议、缺少证据的判断都会被降级或隐藏。

漏报控制的重点是避免只看表层语法。系统会把文件状态、变更规模、风险关键词、历史 PR 摘要和 AI 总结组合起来，让模型能识别“看起来只是小改动，但影响关键链路”的情况。对于后端项目，后续会增加更细的语言规则，例如 Spring Security 配置、Controller 入参校验、MyBatis SQL、事务注解、Redis 缓存键、RocketMQ 消费幂等等。

## 响应速度优化

- 后端限制发送给 DeepSeek 的文件数量。
- 单个 patch 会按固定长度截断，避免大 PR 请求过慢。
- 风险识别和 Review 建议会优先分析高风险文件。
- 前端将 GitHub 数据获取、总结、风险识别和 Review 建议拆成独立按钮，用户可以按需触发，避免一次性等待所有 AI 调用。
- 历史 PR 对比默认不自动执行，只有用户需要上下文对比时才调用更大的模型上下文。

## 模型选择说明

本项目选择 DeepSeek Chat Completions 兼容接口，原因是：

- 接口形式接近常见 Chat Completions，便于在 Next.js Route Handler 中封装。
- 支持 JSON 输出约束，适合生成结构化总结、风险和 Review 建议。
- 对代码 diff、配置变更和中文说明有较好的处理能力。
- 服务端集中调用，便于后续替换为其他兼容模型或多模型交叉验证。
- 普通分析请求显式使用低随机性参数，降低同一 PR 多次分析时的结果波动。

当前设计把“普通 PR 分析”和“历史上下文分析”拆成两套可配置模型。普通分析可以选择响应更快、成本更低的模型；历史对比、跨 PR 影响分析等需要更长上下文的任务，可以使用单独的 `DEEPSEEK_CONTEXT_API_KEY` 和 `DEEPSEEK_CONTEXT_MODEL`。这样既能控制日常使用成本，也为后续接入更大上下文模型或多模型校验留下空间。

## 上下文获取方式

系统的上下文来自三层：GitHub PR 原始上下文、系统预处理上下文、本地历史分析上下文。普通分析默认只使用当前 PR 相关上下文；历史对比分析需要用户手动开启，才会额外使用同仓库历史记录。

上下文处理遵循“分层、筛选、可回看”的原则：

- **第一层：当前 PR 原始上下文**，保证模型知道本次变更的基本事实。
- **第二层：系统预处理上下文**，通过文件类型、风险信号和变更规模筛选重点内容。
- **第三层：AI 结构化结果上下文**，让 Review 建议复用变更总结和风险识别结果，避免重复分析。
- **第四层：同仓库历史上下文**，只在用户手动开启历史对比时使用，用来判断当前 PR 与历史 PR 的关系。
- **第五层：用户反馈上下文**，未来会记录用户对 AI 建议的采纳、忽略、误报标记，用于调整后续建议优先级。

### 1. GitHub PR 原始上下文

前端只提交 GitHub PR URL，后端解析出 `owner`、`repo` 和 PR 编号后，通过 GitHub REST API 获取当前 PR 的基础信息和 changed files。

原始上下文包括：

- PR 标题。
- PR 描述。
- PR 作者。
- PR 状态。
- 创建时间和更新时间。
- 增删行数量。
- changed files。
- 每个文件的状态、增删行、变更规模。
- GitHub 返回的 diff patch。

这些数据是所有 AI 分析的基础。前端不会直接请求 GitHub API，也不会接触 GitHub Token；GitHub Token 只在服务端环境变量中读取。

### 2. 系统预处理上下文

系统不会把完整仓库代码直接发送给模型，也不会无条件把大 PR 的全部 diff 堆进上下文。后端会先对 PR 数据做筛选和整理。

预处理上下文包括：

- 文件状态，例如新增、修改、删除、重命名。
- 文件类型，例如 API、配置、依赖、测试、页面组件。
- 变更规模，例如增删行和 changed files 数量。
- 风险关键词，例如 auth、token、permission、database、env、config。
- 规则预筛选信号，例如鉴权、接口、数据库、配置和测试变化。
- 高风险文件排序。
- patch 截断后的关键片段。

这样做的目的：

- 保留与 Review 最相关的证据。
- 降低大 PR 的模型输入长度。
- 提升响应速度。
- 控制 API 调用成本。
- 降低模型被无关文件干扰的概率。

### 3. 本地历史分析上下文

系统会在浏览器本地保存用户分析过的 PR 摘要，并用 `owner/repo` 作为仓库分类 key。历史记录来自当前浏览器的 `localStorage`，不会上传到服务器、GitHub 或第三方数据库。

历史分析上下文包括：

- 同仓库历史 PR 标题。
- 历史 PR 分析时间。
- 历史 PR 变更规模。
- 历史 AI 总结。
- 历史风险识别结果。
- 历史 Review 建议。
- 截断后的历史 `patchDigest`。
- 用于页面回看的 `changedFileSnapshots`。

历史记录不保存完整仓库代码。`changedFileSnapshots` 用于用户在页面中查看这次 PR 的全部变更内容；`patchDigest` 用于模型历史上下文，避免历史对比时把大体积 diff 全量发送给模型。

### 4. 历史对比上下文

历史 PR 对比分析会在用户手动开启后，把当前 PR 分析结果与同仓库历史 PR 摘要一起构造成大上下文。系统会限制历史记录数量，并明确要求模型区分“当前 PR 证据”和“历史 PR 参考”，降低把历史摘要误当成当前 diff 证据的风险。

历史对比分析的上下文规则：

- 只选择同仓库历史记录。
- 排除当前 PR 自己。
- 默认最多使用最近 10 条历史记录。
- 只使用历史摘要、风险、Review 建议和截断后的 `patchDigest`。
- 不发送完整仓库代码。
- 不发送完整历史 diff。
- 默认不启用，需要用户手动切换并点击生成。

历史对比结果主要回答：

- 当前 PR 是否延续或修改了历史 PR 引入的能力。
- 当前 PR 是否可能影响历史功能。
- 当前 PR 是否重复出现类似风险。
- reviewer 应该重点回看哪些历史 PR。
- 当前分析的置信度和上下文限制是什么。

### 5. 上下文大小控制

为了兼顾分析准确性、响应速度和调用成本，系统采用以下控制策略：

- 单个 patch 会按固定长度截断。
- Review 建议只优先使用高风险文件和大改动文件。
- 历史记录最多保留 50 条。
- 历史对比默认最多使用最近 10 条同仓库历史记录。
- 历史记录会保存页面回看所需的变更快照，但历史对比模型上下文只使用摘要和截断后的 `patchDigest`。
- 普通分析和历史对比拆成独立按钮，由用户按需触发。

这种设计可以避免一次性等待所有 AI 调用，也避免把无关上下文传给模型造成误报。

## 未来扩展方向

### 1. 多语言与多技术栈扩展

当前项目已经具备通用的 PR 获取、diff 解析和 AI 分析链路。后续会把风险规则按语言和技术栈拆分，避免只适配前端项目。

计划扩展方向包括：

- Java / Spring Boot：Controller 入参校验、Service 事务边界、Spring Security 权限配置、异常处理、MyBatis 或 JPA 查询风险。
- Node.js / Next.js：Route Handler、服务端环境变量、鉴权中间件、API 输入输出结构。
- 数据库：SQL 变更、索引变更、字段删除、兼容性迁移、慢查询风险。
- Redis：缓存键设计、过期时间、缓存穿透、缓存击穿、数据一致性。
- RocketMQ / 消息队列：消息幂等、重试、死信、消费顺序、事务消息。
- DevOps 配置：CI/CD、Docker、环境变量、依赖升级和部署脚本风险。

实现方式上，会将规则从单一关键词判断升级为“语言识别 + 文件类型识别 + 技术栈规则 + AI 复核”的组合机制。

### 2. 本地存储与持久化升级

当前历史记录保存在浏览器 `localStorage`，适合本地演示和个人轻量使用，但不适合长期保存大量 PR 分析结果。后续需要支持更可靠的持久化方案。

可选方向：

- 本地开发版使用 SQLite，便于零成本保存更多历史记录。
- 团队版使用 MySQL 或 PostgreSQL，支持多人共享、查询和权限控制。
- 对敏感字段做加密或脱敏，避免保存完整密钥、隐私代码或过大的 diff。
- 为历史记录增加索引，例如仓库、分支、PR 编号、风险类型、分析时间。
- 支持定期清理、导出和备份。

持久化升级后，历史 PR 对比可以从“当前浏览器的本地记录”升级为“团队级仓库 Review 记忆”。

### 3. 多模块智能体拆分

后续可以把当前单一 AI 分析流程拆成多个职责清晰的智能体，让每个模块只处理一种任务。

计划拆分为：

- **PR 描述智能体**：负责总结本次变更、影响范围和测试情况。
- **风险识别智能体**：负责定位风险代码，只输出证据和原因。
- **评审反馈智能体**：负责把风险证据转成可提交的 Review 评论。
- **代码改进智能体**：负责给出更具体的修改方案或示例代码。
- **历史上下文智能体**：负责比较同仓库历史 PR，判断是否有重复风险或影响链路。
- **结果校验智能体**：负责检查输出是否有文件位置、证据、置信度和可执行建议。

这种拆分可以降低单次提示词复杂度，也方便后续做并行分析、模型替换和单模块测试。

### 4. 更贴近真实 reviewer 的 UI 工作流

真实 reviewer 通常不是从上到下读完整报告，而是先快速判断 PR 是否值得深入，再聚焦高风险文件和关键建议。因此 UI 后续应继续向 reviewer 工作流靠近。

计划优化方向：

- 首页先展示 PR 基础信息、变更规模、风险总览和测试信号。
- 风险列表支持按严重级别、文件类型、技术栈、是否需要人工确认筛选。
- 文件变更区支持从风险项跳转到对应 diff 片段。
- Review 建议区支持复制单条建议或完整 Markdown。
- 历史记录按仓库归类，并允许只对比用户选中的历史 PR。
- 明确区分“AI 判断”“规则信号”“当前 diff 证据”“历史 PR 参考”。

目标是让页面像一个辅助 reviewer 的工作台，而不是一份只能阅读的静态报告。

### 5. 上下文分层处理增强

后续会进一步把上下文拆成不同层级，并根据任务选择不同上下文组合。

计划分层：

- **事实层**：PR 标题、描述、作者、文件列表、增删行、patch。
- **结构层**：语言、框架、模块、接口、类、函数、配置项。
- **风险层**：规则命中、敏感文件、关键链路、测试缺口。
- **历史层**：同仓库历史 PR 摘要、历史风险、历史修复建议。
- **反馈层**：用户对 AI 建议的采纳、忽略、误报、补充说明。

不同任务使用不同上下文：变更总结优先使用事实层和结构层；风险识别优先使用事实层、结构层和风险层；历史对比再额外加入历史层；后续个性化优化再加入反馈层。

### 6. 增强人机交互与反馈闭环

AI Review 工具不能只单向输出建议，还需要允许用户反馈哪些建议有用、哪些属于误报。后续会增加人机交互能力，让系统逐步适配项目和团队习惯。

计划能力包括：

- 用户可对每条风险标记“有效”“误报”“已处理”“暂不处理”。
- 用户可补充项目背景，例如“这个接口只给内部系统调用”“这个配置只在测试环境使用”。
- 用户可要求 AI 重新生成某一条建议，而不是重跑整个 PR。
- 系统记录用户反馈，用于后续同仓库 PR 的建议排序。
- 团队可以沉淀自定义规则，例如必须补测试、禁止删除鉴权、数据库变更必须说明回滚方案。

长期目标是让工具从“通用 AI 代码评审助手”升级为“理解项目上下文和团队规则的 Review 协作系统”。
